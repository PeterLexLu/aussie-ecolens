# 项目交付生命周期

> 本文档根据项目实际执行过程补录。项目性质为多云基础设施与后端能力外包，客户为澳大利亚原生动物保护公益组织。仓库内 UI 仅用于功能测试和客户 Demo，不代表客户最终产品界面。

## 1. 项目周期

项目主体交付分为 11 周：

| 阶段 | 周期 | 核心活动 | 主要产出 |
| --- | --- | --- | --- |
| 客户需求发现 | 第 1 周 | 产品经理与客户沟通、收集需求、内部可行性会议 | PRD、需求范围、初步风险 |
| 项目规划 | 第 2 周 | 拆分 30 个 User Stories、估算依赖、制定甘特图与分工 | Product Backlog、甘特图、里程碑 |
| 迭代开发 | 第 3–8 周 | 按 User Stories 开发 AWS、GCP、媒体处理和 Demo 验证能力 | 六周开发增量、测试证据 |
| 测试与客户 Demo | 第 9–10 周 | 系统测试、跨云集成测试、向客户 Demo、收集并落实改进 | 缺陷清单、Demo 反馈、发布候选版本 |
| 交付与 UI 供应商交接 | 第 11 周 | 文档、接口、部署和能力边界交付，与客户方 UI 外包团队合作 | 交付包、接口说明、交接记录 |
| 售后维护 | 6 月底交付后持续 | 回应客户问题、修复缺陷、按需进行小范围改进 | 维护记录、补丁与改进项 |

## 2. 第一周：客户需求与 PRD

产品经理直接与客户沟通，理解公益组织在野生动物媒体上传、自动识别、搜索、标签管理、权限和通知方面的需求。

客户沟通后，产品经理组织团队会议：

- 向团队同步客户目标和业务场景。
- 将客户语言转化为系统能力和验收结果。
- 讨论 AWS、GCP、媒体处理与安全方案的可行性。
- 识别跨云集成、视频推理、媒体访问和交付边界风险。
- 确认本团队不负责最终 UI，测试 UI 仅用于 Demo 和功能验证。

第一周由产品经理完成 PRD，并交由团队进行范围和技术可行性 Review。

## 3. 第二周：项目规划

- 将 PRD 拆分为 30 个可跟踪 User Stories。
- 为 User Stories 定义优先级、验收标准、依赖和负责人。
- 规划六周开发顺序，避免前置依赖未完成时并行开发下游能力。
- 制定甘特图和主要里程碑。
- 确定每周一和周五的 Stand-up Meeting 节奏。
- 明确测试、Demo、交付和售后责任。

## 4. 六周开发流程

### 第 3 周：认证、基础设施和上传基础

- Cognito 身份认证。
- API Gateway 与 Lambda 基础接口。
- S3、DynamoDB 和上传流程。
- 建立测试 UI 的登录与上传入口。

### 第 4 周：异步处理与图片推理

- S3 事件、SQS 和处理 Lambda。
- AWS 到 Cloud Run 的调用契约。
- 图片模型推理、标签和缩略图。

### 第 5 周：视频处理与结果持久化

- OpenCV 视频抽帧。
- 帧级推理和视频标签聚合。
- 结果、所有者与查询数据写入 DynamoDB。

### 第 6 周：媒体管理与查询

- My Media 和 Shared Media 数据能力。
- species、tag-count、file 和 thumbnail 查询。
- 受保护的原媒体和缩略图访问。

### 第 7 周：权限、标签、删除与通知

- Cognito 身份驱动的所有权校验。
- 手动标签和批量操作。
- 所有者删除。
- SNS 物种订阅和邮件通知。

### 第 8 周：可靠性与开发收敛

- checksum 重复上传检测。
- SQS 重试、超时和并发配置。
- 图片与视频兼容性验证。
- 处理状态、异常反馈和跨云联调。

## 5. Stand-up Meeting 节奏

开发阶段每周举行两次 Stand-up Meeting：

### 周一：计划会议

- 回顾 Product Backlog 和当前依赖。
- 选择本周需要完成的 User Stories。
- 明确每条 Story 的负责人、Reviewer 和验收方式。
- 识别当周风险和跨成员协作事项。

### 周五：结算会议

- 逐条检查本周 User Stories 的完成和验收状态。
- 演示已完成能力或测试结果。
- 记录未完成原因、阻塞和后续行动。
- 更新甘特图、Backlog 和下一周候选范围。

Stand-up 的目标不是汇报工时，而是围绕 User Story 做承诺、验收和风险处理。

## 6. 两周测试与客户 Demo

### 第 9 周：系统测试

- 后端和集成自动化测试。
- 图片、视频、搜索和通知的端到端验证。
- 未认证访问、跨用户编辑和删除等权限测试。
- Cloud Run 失败、SQS 重试和重复消息等异常测试。
- 整理客户 Demo 环境和测试样本。

### 第 10 周：客户 Demo 与改进

- 向客户演示完整业务能力，而非把测试 UI 作为最终视觉方案展示。
- 记录客户反馈并区分缺陷、范围内改进和新需求。
- 评估反馈的技术影响、优先级和交付风险。
- 完成双方确认的改进并执行回归测试。

## 7. 一周交付与外部 UI 团队交接

第 11 周完成正式交付，并与客户方 UI 外包团队合作：

- 说明 API、认证方式、请求与响应结构。
- 说明媒体状态、错误码和最终一致性行为。
- 提供测试环境、调用示例和 Demo 流程。
- 说明 My Media、Shared Media、查询、标签、删除和通知的权限边界。
- 明确测试 UI 不属于最终设计资产，UI 团队可以基于接口重新实现正式产品体验。
- 交付部署、环境变量、测试与已知限制说明。

## 8. 售后维护

项目于 6 月底完成交接，随后进入代码售后阶段：

- 回应客户和 UI 供应商在集成过程中的问题。
- 修复可复现的缺陷。
- 对不改变核心范围的小型改进进行评估和实施。
- 新增需求先确认影响、优先级和是否属于原交付范围。
- 为每次售后改动保留问题、原因、验证与发布记录。

## 9. 里程碑

- **M1：** 第一周完成 PRD。
- **M2：** 第二周完成 30 个 User Stories 和甘特图。
- **M3：** 第五周完成图片与视频推理主链路。
- **M4：** 第八周完成开发范围。
- **M5：** 第十周完成测试、客户 Demo 和反馈改进。
- **M6：** 第十一周完成客户与 UI 供应商交接。
- **M7：** 6 月底后进入持续售后维护。
