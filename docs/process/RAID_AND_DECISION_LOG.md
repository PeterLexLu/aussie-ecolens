# RAID 与设计决策记录

> RAID 分别指 Risks、Assumptions、Issues 和 Dependencies。以下内容依据现有架构与实现补录。

## 1. Risks

| ID | 风险 | 影响 | 缓解措施 | 状态 |
| --- | --- | --- | --- | --- |
| R-01 | AWS 与 GCP 请求格式不一致 | 推理链路中断 | 冻结 API Contract，加入集成测试 | 已缓解 |
| R-02 | 视频推理时间长 | 超时或用户误认为失败 | SQS 异步处理、固定频率抽帧、结果聚合 | 已缓解 |
| R-03 | 前端提交伪造 owner | 越权修改或删除 | 后端从 Cognito JWT 获取身份 | 已缓解 |
| R-04 | 重复上传浪费存储与推理资源 | 成本和延迟增加 | 使用 checksum 做平台级去重 | 已缓解 |

## 2. Assumptions

- MVP 用户量和媒体量适合无服务器按需扩展模式。
- 用户已获得上传媒体所需的合法授权。
- 模型输出用于辅助整理，不作为专业物种鉴定的唯一依据。
- 邮件通知依赖用户完成 SNS 订阅确认。
- 测试环境使用的媒体不包含不应公开的敏感信息。

## 3. Dependencies

- AWS Cognito：认证与用户身份。
- API Gateway 与 Lambda：受保护 API 和业务逻辑。
- S3 与 DynamoDB：媒体和元数据。
- SQS 与 SNS：异步处理和通知。
- CloudFront：前端分发。
- Google Cloud Run 与 Cloud Storage：推理服务和模型资源。
- OpenCV：视频抽帧和媒体处理。

## 4. Architecture Decision Records

### ADR-01：采用无服务器架构

**背景：** 上传和推理负载波动明显，团队维护时间有限。

**决策：** 使用 Lambda、S3、DynamoDB、SQS、SNS 和 CloudFront 组成主要应用层。

**取舍：** 减少服务器运维并获得按需扩展能力，但需要处理冷启动、服务限制和分布式诊断问题。

### ADR-02：将机器学习推理部署在 GCP

**背景：** 应用层主要位于 AWS，推理服务需要独立部署和扩展。

**决策：** 使用 Cloud Run 托管推理服务，模型资源存放于 Google Cloud Storage。

**取舍：** 应用与推理可以独立开发，但引入跨云延迟、认证、超时和契约兼容问题。

### ADR-03：在上传与推理之间使用 SQS

**背景：** 视频推理可能超过同步请求可接受时长。

**决策：** S3 事件进入 SQS，由处理 Lambda 消费并调用推理服务。

**取舍：** 提升重试和削峰能力，但前端需要处理最终一致性和处理中状态。

### ADR-04：所有权由后端认证上下文决定

**背景：** 浏览器传入的 owner 字段可被修改。

**决策：** 后端使用 Cognito Token 中的身份执行授权，不信任前端所有权声明。

**取舍：** 增强安全性，同时要求所有受保护接口使用一致的身份中间件。

### ADR-05：使用 checksum 检测重复文件

**背景：** 相同媒体重复上传会造成重复存储和模型推理。

**决策：** 以文件 checksum 建立平台级重复检测索引。

**取舍：** 节省成本并减少等待，但需要定义重复文件的所有权、返回结果和并发写入规则。
